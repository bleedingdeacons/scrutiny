🔍 Scrutiny Plugin

GDPR-compliant audit logging and personal data obscuring for Unity. Scrutiny protects member privacy by masking personal data in the admin UI and recording every access to an audit trail.

What Scrutiny Does

Scrutiny has two jobs. First, it obscures personal data — personal emails and phone numbers are masked with bullet characters for users who don't have permission to see them. Second, it maintains a GDPR audit log that records who viewed, changed, or deleted personal data, when they did it, and from where.

Scrutiny is a required dependency for the Amber plugin. If Scrutiny is deactivated, Amber will automatically shut down to prevent non-compliant handling of personal data.

Requirements

  • WordPress 6.0 or later
  • PHP 8.0 or later
  • Unity plugin — must be installed and activated (provides the member data layer and configuration)
  • Advanced Custom Fields (ACF) — used for the member fields that Scrutiny protects
⚠️
If Unity is not active, you'll see: "This plugin requires the Unity plugin to be installed and activated." Scrutiny will not load until Unity is running.
💡
Scrutiny loads at priority 5 on the unity/loaded hook — before Amber (priority 10). This ensures data obscuring filters are in place before any member data is displayed.

Installation

  1. Make sure the Unity plugin is installed and activated.
  2. Upload the scrutiny folder to your /wp-content/plugins/ directory, or install via the WordPress plugin uploader.
  3. Go to Plugins → Installed Plugins and activate Scrutiny.
  4. On activation, Scrutiny automatically creates its database table (wp_scrutiny_audit_log) and grants the scrutiny_view_personal_data and scrutiny_edit_personal_data capabilities to Administrators.
  5. An Audit Log submenu appears under the Intergroup menu (added by Amber).

How Data Obscuring Works

When a member record is displayed in the admin, Scrutiny intercepts the ACF field values before they are rendered. If the current user does not have the scrutiny_view_personal_data capability, the values are replaced with masked versions:

Email Obscuring

Only the first character of the local part and domain name are shown. The TLD is kept intact.

john@example.com → j•••@e••••••.com

Phone Number Obscuring

Only the last 3 digits are visible. Everything else is replaced with bullet characters.

07700 900123 → •••••••••123

Name Obscuring

Only the first character of each word is shown (available as a method, currently not applied to the admin UI by default).

John Smith → J••• S••••
💡
Every time a personal data field is rendered — whether the value is obscured or shown in full — Scrutiny logs the access to the audit trail. This happens automatically and requires no action from you.

View Permission

Access to unobscured personal data is controlled by the WordPress capability scrutiny_view_personal_data.

Who has it by default?

On activation, Scrutiny grants this capability to the Administrator role only. All other roles see masked values.

Granting it to other roles

If you need other roles (e.g. Editors) to see full personal data, you can assign the capability using a role management plugin such as "Members" or "User Role Editor". Add scrutiny_view_personal_data to the desired role.

🚨
Grant this capability carefully. Under GDPR, access to personal data should follow the principle of least privilege — only users who genuinely need to see the data should have it. Every access is logged in the audit trail.

Edit Permission

The ability to modify personal data values is controlled by a separate capability: scrutiny_edit_personal_data.

How view and edit interact

These two capabilities work together to give fine-grained control over personal data access:

Has View?Has Edit?Behaviour
✅✅Sees the real value and can change it freely.
✅❌Sees the real value but the field is disabled — read-only.
❌✅Sees an obscured placeholder. Can type a new value to replace it, but cannot see the current one.
❌❌Sees an obscured placeholder. Any submitted changes are silently rejected and the existing value is preserved.

Who has it by default?

On activation, Scrutiny grants scrutiny_edit_personal_data to the Administrator role alongside the view capability. Other roles can be granted it via a role management plugin.

Clearing a field value

When a user with edit permission needs to intentionally remove a value (e.g. a member has withdrawn their email), a Clear button is available alongside the field. This sends a special sentinel value so Scrutiny can distinguish an intentional clear from an untouched empty form submission.

⚠️
If a user without the edit capability saves the member form, Scrutiny silently preserves the existing stored value — no data is lost, and no error is shown. This prevents accidental overwrites by users who can see but not change personal data.

What Gets Logged

Scrutiny automatically logs the following events for personal data fields:

Member Events

  • Viewing — when a member edit form is opened in the admin, or when personal data fields are loaded on the front end
  • Updating — when a personal email or mobile number value is changed
  • Deleting — when a member post is moved to Trash or permanently deleted

Group Events

  • Updating — when a group's contact name, email, or phone number is changed
  • Hiding — when a group's post status is set to private (logged as an update to all contact fields)
  • Deleting — when a group is moved to Trash or permanently deleted

Meeting Events

  • Updating — when a meeting's contact name, email, or phone number is changed (tracked as part of the parent group save)
✅
No raw personal data is stored in the audit log. Entries record only the field name (e.g. "Personal Email"), the member ID, the WordPress user who performed the action, and a timestamp. The actual email address or phone number is never written to the log.

Action Types

Each audit log entry has a colour-coded action badge:

ActionMeaning
ViewA personal data field was displayed to a user (in admin or on the front end).

UpdateA personal data field value was changed.

Navigate to Intergroup → Audit Log (Administrators only). The page shows a paginated table of all audit entries, newest first.

Each row displays:

💡
Use the Refresh button (next to the page title) to reload the audit log without resetting your filters. This is useful for monitoring access in real time.

Purging Old Entries

Under GDPR Article 5(1)(e), personal data audit logs should not be kept longer than necessary. At the bottom of the Audit Log page, you'll find a Data Retention section with a purge button.

How Purging Works

  1. Click "Purge entries older than 1 year" at the bottom of the Audit Log page.
  2. Confirm the action in the browser dialog that appears.
  3. All audit log entries older than 365 days are permanently deleted from the database.
  4. The purge action itself is recorded as a new audit entry so there is a record of when data was cleaned up.
🚨
Purging is permanent and irreversible. Deleted entries cannot be recovered. Only Administrators can perform a purge.

IP Anonymisation

To comply with GDPR, Scrutiny does not store full IP addresses. Instead, it anonymises them before writing to the database:

  • IPv4 — the last octet is replaced with 0 (e.g. 192.168.1.42 → 192.168.1.0)
  • IPv6 — the last 80 bits are zeroed out, storing only the subnet prefix

This stores enough information to identify a general network location for security purposes, without pinpointing an individual device.

Troubleshooting

"Scrutiny requires the Unity plugin" error on activation

Go to Plugins → Installed Plugins, find Unity, and activate it. Then try activating Scrutiny again.

I can't see the Audit Log menu

The Audit Log appears under Intergroup → Audit Log. It requires the manage_options capability, which only Administrators have by default. If you don't see the Intergroup menu at all, check that Amber is also activated.

Personal data fields aren't being masked

Check that your user account has the scrutiny_view_personal_data capability. If you do, you'll see unmasked values — that's expected. Log in with a non-administrator account to verify that masking is working. Note that a separate scrutiny_edit_personal_data capability controls whether the field can be changed — users with view but not edit will see the real value in a disabled (read-only) input.

Audit log table is empty

If you just activated Scrutiny, the table starts empty. Open a member edit page and the first "View" entries will appear. If the table was created but no entries appear after editing members, check the WordPress error log for database write errors.

Amber was deactivated when I deactivated Scrutiny

This is by design. Amber requires Scrutiny for GDPR compliance and will automatically deactivate itself if Scrutiny is not running. Reactivate Scrutiny first, then reactivate Amber.

FAQ

Does the audit log store actual personal data?

No. The log records the field name (e.g. "Personal Email", "Group Contact Phone"), the entity post ID, the WordPress user who accessed it, and a timestamp. The actual email address, phone number, or contact name is never written to the log table.

How often should I purge the audit log?

GDPR doesn't specify an exact retention period — it should be "no longer than necessary". The built-in purge button removes entries older than 1 year, which is a reasonable default for most organisations. You may want to purge more frequently if you have many users accessing member data.

Can I export the audit log?

There is no built-in export function in the current version. You can query the wp_scrutiny_audit_log database table directly if you need to produce reports for a GDPR subject access request or internal audit.

What happens to the audit log table if I deactivate Scrutiny?

The database table and its data are preserved when you deactivate or reactivate the plugin. No data is deleted on deactivation.

Can I use Scrutiny without Amber?

Yes. Scrutiny operates independently — it hooks into Unity's member system directly. Amber depends on Scrutiny, not the other way around. If you only need data obscuring and audit logging, Scrutiny works on its own.

Who can access the Audit Log page?

Only users with the manage_options capability — typically Administrators only. This is separate from scrutiny_view_personal_data (which controls who can see unmasked field values) and scrutiny_edit_personal_data (which controls who can change them).
